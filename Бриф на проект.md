## Описание проекта

## 1. Общие сведения
Целью проекта является создание масштабируемого ML-сервиса, предоставляющего пользователям возможность выполнения предсказаний на основе загружаемых моделей с автоматическим списанием внутренней валюты (кредитов) за каждый успешный запрос.

## 2. Функциональные требования
1.  **Пользовательский блок:** Регистрация, аутентификация (JWT), управление личным кабинетом.
2.  **ML-блок:** Загрузка моделей (Scikit-learn), выполнение предсказаний через API, асинхронная обработка запросов.
3.  **Биллинговый блок:** Учет баланса кредитов, списание средств по факту успешного выполнения задачи, интеграция с платежным шлюзом (для пополнения баланса).
4.  **Аналитический блок:** Дашборд (Streamlit) для отображения статистики использования сервиса и расхода кредитов.
5.  **API:** REST API с автоматической документацией (Swagger/OpenAPI).

## 3. Технологический стек
*   **Язык:** Python.
*   **ML:** Scikit-learn.
*   **БД:** PostgreSQL.
*   **Контейнеризация:** Docker, Docker Compose.
*   **Мониторинг:** Prometheus + Grafana.
*   **Тестирование:** Unit-тесты (покрытие > 70%).

---

## 4. Основные этапы реализации

1.  **Проектирование:** Разработка архитектуры, проектирование схемы БД, описание API.
2.  **Backend-разработка:** Реализация системы аутентификации и базового REST API.
3.  **ML-интеграция:** Реализация сервиса предсказаний с использованием асинхронной очереди задач (например, Celery + Redis).
4.  **Биллинг:** Разработка логики транзакций (списание/начисление кредитов).
5.  **Интерфейс:** Создание дашборда на Streamlit для пользователей.
6.  **Инфраструктура и мониторинг:** Настройка Docker-контейнеров, подключение Prometheus/Grafana.
7.  **Тестирование и документация:** Написание тестов и финальной документации.

## 5. Вариативные части: Маркетинговые механики биллинга

Студентам предлагается реализовать одну или несколько механик стимулирования пользователей. Выбор механики влияет на сложность проектирования базы данных и логики обработки транзакций.

- **Вариант А (Базовый: Система промокодов):**
    
    - Реализация функционала создания и активации промокодов.
    - Промокод может давать фиксированную сумму кредитов (например, «WELCOME100» дает 100 кредитов) или процентную скидку на следующее пополнение баланса.
    - _Задача:_ Реализовать проверку срока действия промокода и ограничение на количество активаций (одноразовые/многоразовые).
- **Вариант Б (Продвинутый: Динамические скидки и уровни лояльности):**
    
    - Реализация системы «уровней» пользователя (например: Bronze, Silver, Gold).
    - Чем больше предсказаний совершил пользователь за месяц, тем выше его уровень, который дает автоматическую скидку на стоимость каждого предсказания (например, -5%, -10%, -20%).
    - _Задача:_ Реализовать фоновую задачу (cron), которая раз в месяц пересчитывает статус пользователя на основе статистики его активности.
- **Вариант В (Экспертный: Реферальная программа и «Кэшбэк»):**
    
    - Реализация реферальной системы: пользователь получает уникальную ссылку; если по ней регистрируется новый клиент, оба получают бонусные кредиты.
    - Реализация системы «Кэшбэк»: возврат части потраченных кредитов на бонусный счет при достижении определенного порога трат за неделю.
    - _Задача:_ Проектирование сложной системы учета транзакций, где необходимо разделять «основной баланс» (купленные кредиты) и «бонусный баланс» (полученные по акциям), с правилами их приоритетного списания.

---

### Советы по реализации для студентов:

1. **Приоритетность списания:** Если вы выберете вариант с бонусными кредитами, продумайте логику: что списывается первым — купленные кредиты или подарочные? (Обычно сначала списываются бонусы, чтобы стимулировать их использование).
2. **Безопасность:** Любая акция — это потенциальная уязвимость. Убедитесь, что пользователь не может активировать один и тот же промокод несколько раз, используя баги в API.
3. **Масштабируемость:** При реализации скидок старайтесь не «зашивать» логику в код (hardcode). Лучше вынести правила скидок в отдельные таблицы БД, чтобы ими можно было управлять через административную панель без пересборки проекта.

## 6. Критерии приемки (Чек-лист)
*   [ ] Система аутентификации работает корректно (JWT, роли пользователей).
*   [ ] Биллинговая подсистема корректно списывает кредиты (атомарность операций).
*   [ ] ML-модель работает асинхронно (не блокирует API).
*   [ ] Наличие Swagger-документации для всех эндпоинтов.
*   [ ] Проект запускается одной командой `docker-compose up`.
*   [ ] Настроен мониторинг (визуализация метрик в Grafana).
*   [ ] Покрытие кода тестами > 70%.
*   [ ] Наличие краткого бизнес-плана (УТП, описание финмодели).
