# Business Plan

## 1. Идея проекта
`Apartment Price Service` — это ML-сервис для оценки стоимости недвижимости по набору признаков объекта. Пользователь регистрируется, пополняет внутренний баланс кредитов, отправляет запрос на предсказание и получает результат через API.

Основная ценность сервиса:
- быстрый расчет стоимости жилья по API;
- прозрачная модель оплаты за фактическое использование;
- удобная история запросов и расходов;
- возможность интеграции в proptech-сценарии и аналитические инструменты.

## 2. УТП
- `Pay-per-use` модель: пользователь платит только за фактические предсказания.
- Асинхронная обработка: API не блокируется во время выполнения ML-задачи.
- Прозрачный биллинг: каждый расход и каждое начисление фиксируются в журнале транзакций.
- Готовность к масштабированию: PostgreSQL, Celery, Redis, Docker Compose, Prometheus, Grafana.

## 3. Целевая аудитория
- частные пользователи, которые хотят быстро оценить жилье;
- риелторы и агентства недвижимости;
- аналитики рынка недвижимости;
- небольшие proptech-команды и стартапы;
- команды, которым нужен API-слой для оценки недвижимости.

## 4. Сценарий использования
1. Пользователь регистрируется в системе.
2. Получает стартовый бонус или пополняет баланс.
3. Отправляет параметры объекта на оценку.
4. Запрос уходит в очередь и обрабатывается ML-воркером.
5. После успешного выполнения пользователь получает результат.
6. С баланса списываются кредиты только по факту успешного предсказания.

## 5. Финансовая модель
Базовая модель монетизации:
- пользователь покупает внутренние кредиты;
- каждое предсказание стоит фиксированное число кредитов;
- сервис стимулирует первое использование через промокоды.

Пример:
- стартовый welcome-бонус;
- пополнение на 100, 500, 1000 кредитов;
- фиксированная стоимость одного предсказания, например 10 кредитов.

## 6. Маркетинговая механика
В проекте выбран `Вариант А — промокоды`.

Что реализуется:
- создание промокодов;
- ограничение количества активаций;
- проверка срока действия;
- запрет повторной активации одним пользователем;
- начисление бонусных кредитов через журнал транзакций.

Промокоды подходят для:
- онбординга новых пользователей;
- маркетинговых акций;
- демонстрации сервиса преподавателю на защите.

## 7. Потенциальные источники роста
- подключение реального платежного шлюза;
- поддержка нескольких ML-моделей и версий;
- более точные модели на датасетах квартирного уровня;
- личные тарифы для риелторов и B2B-клиентов;
- кабинет администратора для управления промокодами и моделями.

## 8. Основные риски
- низкое качество датасета;
- сложность поддержки биллинга при росте числа сценариев;
- повторная обработка асинхронных задач;
- расхождение между ML-входом и API-контрактом;
- нехватка времени на тестирование и документацию.

Как риски снижаются:
- простая маркетинговая механика без перегруженного домена;
- журнал транзакций для аудита;
- асинхронный pipeline с явными статусами задач;
- мониторинг и дашборды;
- unit-тесты на ключевую бизнес-логику.

## 9. Почему проект жизнеспособен как учебный MVP
- закрывает сразу несколько инженерных компетенций: backend, ML, async, DB, Docker, observability;
- имеет понятную бизнес-логику и монетизацию;
- достаточно компактен, чтобы успеть реализовать за учебный срок;
- легко демонстрируется через Swagger, Grafana и Streamlit.
